# UI V2 全生命周期设计

## 1. 背景

发布工具当前前端已经积累了大量历史能力：

- 快速发布
- 常规发布
- 批量发布
- 账号管理
- 图床设置
- 偏好设置
- 多平台配置

但这些能力分散在不同页面、不同入口和不同组件体系中。用户从“我要发文章”到“我要调整设置”，体验路径并不统一。工程上也存在问题：

- 入口与配置读取路径不统一
- UI 与业务桥接边界不清晰
- 旧设置表单高度离散
- 没有全局迁移路线图
- 存在 `siyuan/` 真实插件运行时与 `src/` iframe SPA 双系统

本设计的目标不是一口气推翻旧系统，而是把 V2 作为一个完整的渐进迁移项目来设计和落地。

## 2. 设计目标

## 2.0 最高优先级规则

V2 的最高优先级实现规则如下：

- 优先使用思源笔记原生 UI
- 优先复用宿主已注入的 `siyuan-style.css`
- 优先复用宿主已有的菜单、按钮、表单、状态反馈和布局能力
- 只有宿主不存在合适能力时，才允许局部自定义样式

这条规则优先级高于“视觉还原度”本身。

原因：

- 思源本质是 Electron 宿主应用
- V2 运行在宿主上下文中
- 复用原生 UI 能显著降低冲突与维护成本
- 复用原生主题能力可以减少大量自定义组件工作量

更进一步，它的重要性体现在：

1. 让用户进入插件时更有 `feel at home` 的体验
2. 明显减少样式体积，省去重复定义大量通用样式
3. 让其他扩展和未来能力接入更容易兼容
4. 自动适配宿主主题、主图和模式切换
5. 在桌面、移动、浅色、深色等环境中更容易做到无缝衔接

因此，在后续每一个里程碑中，都应优先回答：

> 当前能力是否可以直接建立在思源原生 UI 之上？

### 2.1 产品目标

- 用户进入发布工具时，首先看到的是与当前文档最相关的快速发布主界面
- 用户在需要设置时，不跳出到另一套完全不同的产品框架，而是在同一工作壳内展开设置
- 用户能在同一套心智模型下完成：
  - 快速发布
  - 新增账号
  - 图床设置
  - 偏好设置
  - 平台配置维护

### 2.2 工程目标

- 新旧 UI 共存，支持回退
- 配置格式保持兼容
- 发布逻辑复用
- 平台配置逻辑尽量桥接，而不是立即重写
- 各里程碑都有清晰输出和验收标准
- V2 新能力全部基于真实 DOM 挂载
- 逐步减少并最终替换 iframe SPA 依赖

### 2.3 迁移目标

- 在早期版本中优先交付统一入口和快速发布主路径
- 在中期版本中逐步吸收设置能力
- 在后期版本中收敛旧页面与表单抽象

## 3. 非目标

- 不在首轮重构中重写所有平台适配器
- 不在首轮重构中改变平台配置格式
- 不在未稳定前移除全部旧设置页面
- 不以“视觉改版”为名顺手解决所有历史页面问题

## 4. 设计输入与取法

### 4.1 Ant Design

采用其设计价值观中的以下原则：

- 自然：让主任务路径更顺畅
- 确定性：让入口、状态、配置来源更清晰
- 生长性：让 V2 可逐步替换旧系统，而不是一次性重做

### 4.2 飞书式体验

本项目对“飞书风格”的取法不是像素级模仿，而是：

- 弱装饰、强层级
- 列表优先、详情后置
- 在同一工作面内做渐进展开
- 让用户始终知道自己当前处于什么层级

注意：

- “飞书式体验”只用于指导信息架构、层级和交互取向
- 不应凌驾于“思源原生 UI 优先”之上
- 也就是说，应优先做“像思源原生应用一样自然的工具体验”，而不是先做一套独立设计系统

### 4.3 用户补充设计图

补充截图说明：

- 需要一个带品牌识别的工作面
- 快速发布与设置不应割裂成两套产品
- 设置态存在清晰的导航区和内容区
- 图床设置需要支持分组表单结构

## 5. 目标交互模型

### 5.1 统一工作壳

V2 必须采用 `UnifiedWorkspaceShell`，由以下区域构成：

- 品牌视觉区
- 导航区
- 主内容区
- 详情区

这些区域不是在所有状态下都完全展开，而是根据当前任务动态显隐。

### 5.2 主界面态

主界面态是 V2 的默认状态，聚焦“当前文档的快速发布”。

特征：

- 主内容区聚焦当前文档和已配置平台
- 导航区可隐藏、折叠或弱化
- 详情区默认隐藏
- 设置入口存在，但不是主视觉中心

### 5.3 设置展开态

设置展开态与主界面态属于同一工作壳，仅仅是展开更多导航和细节。

特征：

- 导航区展开
- 主内容区切换到列表、分组表单或桥接视图
- 详情区可根据平台配置流程展开
- 用户不离开统一工作壳

## 7. 技术架构

### 7.0 历史架构说明

当前仓库存在两套历史运行形态：

- `siyuan/`：真实插件入口、事件、菜单、宿主交互
- `src/`：被 iframe 挂载的 SPA 页面与路由系统

这解释了当前为何会出现：

- 双入口
- 双运行时
- 双存储读取习惯
- 业务逻辑散落在 SPA 页面中

V2 的总体方向必须是：

> 让真实 DOM 挂载逐步取代 iframe SPA，而不是继续让两套运行形态长期并存。

### 7.0.1 构建链说明

当前项目还存在历史构建分工：

- 插件链路保留旧脚本和旧打包方式
- SPA 链路使用现有 `vite.config.ts`

V2 迁移阶段需要独立构建链，原因是：

- 不能立即破坏现有 V1 构建和发布
- 需要给 V2 DOM-only 路径提供单独测试通道
- 需要在收敛旧系统前允许新系统单独 `dev/build/preview`

因此 V2 应新增独立构建入口，例如：

- `vite.v2.config.ts`
- `dev:v2`
- `build:v2`

该链路应直接构建 V2 的插件态入口，而不是再引入新的 html 页面。它的目标是服务迁移，不是永久增加一套新技术债。待 V2 稳定后，应把旧链路逐步收敛。

### 7.1 V2 Host

负责：

- 创建挂载容器
- 控制打开关闭
- 管理初始化失败
- 提供统一回退到旧 UI 的能力

要求：

- 只能有一个正式运行时入口
- 禁止在多个插件入口文件中散写挂载逻辑
- 必须服务于真实 DOM 挂载，而不是再次包一层 iframe

### 7.2 UnifiedWorkspaceShell

负责：

- 品牌视觉区布局
- 导航区布局
- 主内容区布局
- 详情区布局
- 区域显隐逻辑

要求：

- 快速发布和设置展开态都必须基于它
- 不能出现“主界面一个壳、设置一个壳”的双框架

### 7.3 Configuration Gateway

负责：

- `useV2UI`
- 入口相关偏好
- 文档菜单相关偏好
- 与 V2 主入口行为有关的配置读取

要求：

- 单一真相源
- 禁止直接在入口层读取 `window.localStorage`

### 7.4 状态层

V2 只新增 UI 状态，不复制业务配置。

建议状态切分：

- `useV2UIStore`
  - 主视图
  - 设置分类
  - 详情区开关
  - 当前选中项
- `useV2PublishStore`
  - 发布中状态
  - 发布结果
  - 重试状态
- `useV2SettingsStore`
  - 设置页面级上下文
  - 当前桥接对象

### 7.5 业务能力复用

继续复用：

- 平台配置存储
- 平台列表解析
- 发布逻辑
- 文档上下文读取
- 平台元数据
- 宿主原生 UI 与 `siyuan-style.css`

注意：

- 复用业务逻辑是允许的
- 继续把旧 SPA 页面整体塞进 iframe，不应成为 V2 的长期实现方式
- 对 UI 而言，优先复用宿主原生样式和交互，不优先新造组件库

### 7.6 平台表单桥接策略

现有平台设置是大量专用组件，不适合首轮统一重写。

因此采用两段式策略：

1. 第一阶段：V2 设置容器桥接旧表单能力
2. 第二阶段：逐步提炼公共字段，演进到 schema 驱动

这里的“桥接”优先指：

- 复用配置解析
- 复用业务提交逻辑
- 复用已有字段定义

而不是长期保留对旧 iframe 页面本身的依赖。

## 8. 全生命周期路线图

### Milestone 0: 入口与治理基座

#### 目标

建立能运行、能回退、能扩展的 V2 基座。

#### 范围

- 统一入口定义
- `useV2UI` 开关
- `V2Host`
- 单一配置源
- 回退机制
- 明确 V2 主路径只允许真实 DOM 挂载
- 建立 V2 独立构建入口用于验证 DOM-only 路径

#### 输出

- 稳定可开的 V2 入口
- 失败自动回退
- 最小可运行 `V2App`

#### 退出条件

- 开关关：旧 UI
- 开关开：V2 Host
- Host 失败：旧 UI
- 关键偏好读取路径唯一

### Milestone 1: 样式系统与工作壳骨架

#### 目标

建立 V2 样式基础，为后续主界面和设置展开态提供统一视觉骨架。

#### 范围

- `.syp-v2` 命名空间
- 设计令牌
- 基础排版、按钮、卡片
- `UnifiedWorkspaceShell` 骨架样式

#### 输出

- 样式统一入口
- 品牌区 / 导航区 / 内容区 / 详情区骨架

#### 退出条件

- V2 样式不污染旧 UI
- 统一壳骨架可渲染
- 样式不再双轨

### Milestone 2: 快速发布主界面

#### 目标

交付主界面态，让 V2 第一屏真正成为快速发布界面。

#### 范围

- 当前文档信息展示
- 已配置平台列表
- 空状态
- 设置展开入口

#### 输出

- 主界面态可展示真实平台
- 用户可从主界面切到设置展开态

#### 退出条件

- 已配置态和空状态都可工作
- 不需要进入设置即可完成平台选择动作准备

### Milestone 3: 发布动作闭环

#### 目标

打通主界面态的发布闭环。

#### 范围

- 单平台发布
- 发布中状态
- 发布成功/失败反馈
- 重试

#### 输出

- V2 主界面态完成真实发布

#### 退出条件

- 用户可以在 V2 里完成当前文档发布
- 失败不阻断后续操作

### Milestone 4: 设置展开态第一阶段

#### 目标

在统一工作壳内完成设置展开态的第一阶段能力。

#### 范围

- 账号列表
- 平台选择
- 图床设置内容区
- 偏好设置内容区
- 至少一类平台配置桥接
- 开始把高频设置能力从 iframe SPA 中抽离

#### 输出

- 统一工作壳展开态可用
- 设置分类可切换
- 部分桥接链路可跑通

#### 退出条件

- 新增账号流程可用
- 图床设置可展示与保存
- 偏好设置可展示与保存

### Milestone 5: 设置展开态第二阶段

#### 目标

扩展设置展开态，逐步替换高频旧设置能力。

#### 范围

- 更多平台配置桥接
- 设置态交互打磨
- 图床和偏好高频能力稳定化

#### 输出

- 高频设置路径优先完成迁移

#### 退出条件

- 大多数用户高频设置场景不再依赖旧设置页

### Milestone 6: 收敛与稳定发布

#### 目标

完成 V2 与旧 UI 的长期共存策略、收敛策略和稳定发布策略。

#### 范围

- 统计仍依赖旧 UI 的能力
- 判断哪些旧入口可以下沉或隐藏
- 准备后续废弃计划
- 统计仍依赖 iframe SPA 的能力
- 制定 iframe 退役清单

#### 输出

- V2 稳定发布策略
- 旧 UI 收敛清单

#### 退出条件

- 至少一个稳定版本周期内无阻断性问题
- 回退路径仍然可用
- 旧 UI 是否废弃有明确判据

## 9. 渐进迁移策略

### 9.1 共存策略

- 默认允许旧 UI 与 V2 共存
- V2 通过开关启用
- 旧入口不立即删除
- iframe 路径仅作为历史兼容路径，不能继续承载 V2 新能力

### 9.2 桥接策略

- 能桥接就桥接
- 不能桥接的先保留旧页
- 不用“一定全新重写”绑死后续阶段
- 但桥接优先到真实 DOM 组件层，不优先到 iframe 页面层

### 9.3 回退策略

以下情况可直接回退到上一个稳定点：

- 入口不稳定
- 配置双读
- V2 影响旧流程
- 某一阶段的实现破坏前置阶段验收

## 10. 验证策略

### 10.1 每个里程碑都必须具备

- 手工 smoke 清单
- 至少一项自动化或半自动验证
- 回退验证

### 10.2 重点验证对象

- 顶栏入口
- 文档快捷入口
- 偏好开关
- 主界面平台展示
- 发布动作
- 账号设置
- 图床设置
- 偏好设置

## 11. 风险

### 风险 1：仍然按局部思维推进

结果会是：

- 某个里程碑优化局部
- 后续阶段再推翻
- 总体成本升高

### 风险 2：过早重写所有平台表单

结果会是：

- 开发周期极长
- 频繁返工
- 主路径迟迟不能交付

### 风险 3：缺少生命周期管理

结果会是：

- 每一阶段都像一个临时 patch
- 维护者无法判断接下来该做什么

### 风险 4：继续维护 iframe 双系统

结果会是：

- DOM 版本和 iframe 版本长期重复演进
- 存储和入口逻辑持续分叉
- 维护成本继续上升

## 12. 决策

- V2 必须按全生命周期方案推进
- 仍然保留强门禁：前一里程碑未通过，禁止进入下一里程碑实现
- 优先交付主路径，再逐步吸收设置能力
- V2 的长期方向必须是 DOM-only，iframe 仅作为过渡兼容层
- 允许慢，但不允许乱
