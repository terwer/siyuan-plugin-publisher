# UI V2 全生命周期设计

## 1. 背景

发布工具当前前端已经积累了大量历史能力：

- 快速发布
- 常规发布
- 批量发布
- 账号管理
- 图床设置
- 偏好设置
- 多平台配置

但这些能力分散在不同页面、不同入口和不同组件体系中。用户从“我要发文章”到“我要调整设置”，体验路径并不统一。工程上也存在问题：

- 入口与配置读取路径不统一
- UI 与业务桥接边界不清晰
- 旧设置表单高度离散
- 没有全局迁移路线图
- 存在 `siyuan/` 真实插件运行时与 `src/` iframe SPA 双系统

本设计的目标不是一口气推翻旧系统，而是把 V2 作为一个完整的渐进迁移项目来设计和落地。

## 2. 设计目标

## 2.0 最高优先级规则

V2 的最高优先级实现规则如下：

- 优先使用思源笔记原生 UI
- 优先复用宿主已注入的 `siyuan-style.css`
- 优先复用宿主已有的菜单、按钮、表单、状态反馈和布局能力
- 只有宿主不存在合适能力时，才允许局部自定义样式

这条规则优先级高于“视觉还原度”本身。

原因：

- 思源本质是 Electron 宿主应用
- V2 运行在宿主上下文中
- 复用原生 UI 能显著降低冲突与维护成本
- 复用原生主题能力可以减少大量自定义组件工作量

更进一步，它的重要性体现在：

1. 让用户进入插件时更有 `feel at home` 的体验
2. 明显减少样式体积，省去重复定义大量通用样式
3. 让其他扩展和未来能力接入更容易兼容
4. 自动适配宿主主题、主图和模式切换
5. 在桌面、移动、浅色、深色等环境中更容易做到无缝衔接

因此，在后续每一个里程碑中，都应优先回答：

> 当前能力是否可以直接建立在思源原生 UI 之上？

### 2.1 产品目标

- 用户进入发布工具时，首先看到的是与当前文档最相关的快速发布主界面
- 用户在需要设置时，不跳出到另一套完全不同的产品框架，而是在同一工作壳内展开设置
- 用户能在同一套心智模型下完成：
  - 快速发布
  - 新增账号
  - 图床设置
  - 偏好设置
  - 平台配置维护

### 2.2 工程目标

- 新旧 UI 共存，支持回退
- 配置格式保持兼容
- 发布逻辑复用
- 平台配置逻辑尽量桥接，而不是立即重写
- 各里程碑都有清晰输出和验收标准
- V2 新能力全部基于真实 DOM 挂载
- 逐步减少并最终替换 iframe SPA 依赖

### 2.3 迁移目标

- 在早期版本中优先交付统一入口和快速发布主路径
- 在中期版本中逐步吸收设置能力
- 在后期版本中收敛旧页面与表单抽象

## 3. 非目标

- 不在首轮重构中重写所有平台适配器
- 不在首轮重构中改变平台配置格式
- 不在未稳定前移除全部旧设置页面
- 不以“视觉改版”为名顺手解决所有历史页面问题

## 4. 设计输入与取法

### 4.1 Ant Design

采用其设计价值观中的以下原则：

- 自然：让主任务路径更顺畅
- 确定性：让入口、状态、配置来源更清晰
- 生长性：让 V2 可逐步替换旧系统，而不是一次性重做

### 4.2 飞书式体验

本项目对“飞书风格”的取法不是像素级模仿，而是：

- 弱装饰、强层级
- 列表优先、详情后置
- 在同一工作面内做渐进展开
- 让用户始终知道自己当前处于什么层级

注意：

- “飞书式体验”只用于指导信息架构、层级和交互取向
- 不应凌驾于“思源原生 UI 优先”之上
- 也就是说，应优先做“像思源原生应用一样自然的工具体验”，而不是先做一套独立设计系统

### 4.3 用户补充设计图

补充截图说明：

- 需要一个带品牌识别的工作面
- 快速发布与设置不应割裂成两套产品
- 设置态存在清晰的导航区和内容区
- 图床设置需要支持分组表单结构

## 5. 目标交互模型

### 5.1 统一工作壳

V2 必须采用 `UnifiedWorkspaceShell`，由以下区域构成：

- **导航区**：设置展开态时展示，提供设置分类切换（账号/图床/偏好）
- **主内容区**：承载当前视图的具体内容

品牌视觉区（header，含 chip、标题、操作按钮）由 `V2App.vue` 独立管理，不在 `UnifiedWorkspaceShell` 内部。详情区为未来预留，当前版本暂不实现。

各区域根据当前任务动态显隐：快速发布态隐藏导航区，设置展开态展示导航区。

### 5.2 主界面态

主界面态是 V2 的默认状态，聚焦“当前文档的快速发布”。

特征：

- 主内容区聚焦当前文档和已配置平台
- 导航区可隐藏、折叠或弱化
- 详情区默认隐藏
- 设置入口存在，但不是主视觉中心

### 5.3 设置展开态

设置展开态与主界面态属于同一工作壳，仅仅是展开更多导航和细节。

特征：

- 导航区展开
- 主内容区切换到列表、分组表单或桥接视图
- 详情区可根据平台配置流程展开
- 用户不离开统一工作壳

## 6. 目标信息架构

### 6.1 顶层状态

- `QUICK_PUBLISH`
- `SETTINGS`

### 6.2 设置分类

- `ACCOUNT`
- `PICBED`
- `PREFERENCE`

### 6.3 设置内部视图

- `ACCOUNT_LIST` / `PLATFORM_SELECT` / `PLATFORM_CONFIG`：账号设置的三层视图流，由 `useV2Settings.accountView` 驱动
- `PICBED`：图床设置单一视图，直接由 `V2PicBedSettings` 组件承载
- `PREFERENCE`：偏好设置单一视图，直接由 `V2PreferenceSettings` 组件承载

## 7. 技术架构

### 7.0 历史架构说明

当前仓库存在两套历史运行形态：

- `siyuan/`：真实插件入口、事件、菜单、宿主交互
- `src/`：被 iframe 挂载的 SPA 页面与路由系统

这解释了当前为何会出现：

- 双入口
- 双运行时
- 双存储读取习惯
- 业务逻辑散落在 SPA 页面中

V2 的总体方向必须是：

> 让真实 DOM 挂载逐步取代 iframe SPA，而不是继续让两套运行形态长期并存。

### 7.0.1 构建链说明

当前项目的历史构建分工：

- 插件链路保留旧脚本和旧打包方式
- SPA 链路使用现有 `vite.config.ts`

V2 不新增独立构建链，而是复用现有 `vite.config.ts` 统一编译。原因：

- V2 组件本质上是 Vue SFC，与现有源码共享同一套编译流程
- 新增独立构建链会引入不必要的技术债和配置维护成本
- 现有构建链已支持条件编译和代码分割，足以服务 V2 迁移

因此 V2 源码以常规组件形式置于 `src/components/v2/*`、`src/composables/v2/*`、`src/assets/v2/*` 等目录下，由现有构建链统一处理。待 V2 稳定、旧 SPA 退役后，再收敛旧页面入口和冗余路由，而不是维护多套构建配置。

### 7.1 V2 Host

负责：

- 创建挂载容器（基于思源原生 `Menu` 创建 `mountPoint`）
- 控制 Vue 应用的 mount / unmount 生命周期
- 管理初始化失败时的资源清理（`close()`）
- 注入宿主 i18n 到 V2 Vue 运行时

不负责：

- 回退到旧 UI 的决策逻辑（由调用方 `topbar.ts` / `widgetInvoke.ts` 通过 try/catch 控制）

要求：

- 只能有一个正式运行时入口
- 禁止在多个插件入口文件中散写挂载逻辑
- 必须服务于真实 DOM 挂载，而不是再次包一层 iframe

### 7.2 UnifiedWorkspaceShell

负责：

- 导航区布局（设置展开态时展示，提供账号/图床/偏好分类切换）
- 主内容区布局（通过 `<slot />` 注入当前视图内容）
- 区域显隐逻辑（根据 `currentView` 控制导航区是否渲染）

不负责：

- 品牌视觉区（由 `V2App.vue` 的 `syp-header` 独立管理）
- 详情区（当前版本预留，未实现）

要求：

- 快速发布和设置展开态都必须基于它
- 不能出现“主界面一个壳、设置一个壳”的双框架

### 7.3 Configuration Gateway

负责：

- `useV2UI`
- 入口相关偏好
- 文档菜单相关偏好
- 与 V2 主入口行为有关的配置读取

要求：

- 单一真相源
- 禁止直接在入口层读取 `window.localStorage`

### 7.4 状态层

V2 只新增 UI 状态，不复制业务配置。

实际采用 Vue Composable 模式管理状态，而非新增 Pinia Store：

- `useV2QuickPublish`
  - 负责：快速发布主界面的平台列表、发布状态、预览链接
  - 内部使用 `reactive` 持有 UI 状态
  - 复用 `usePublish()`、`usePublishConfig()` 等业务逻辑
- `useV2Settings`
  - 负责：设置展开态的导航、账号列表、平台选择、配置桥接上下文
  - 内部使用 `reactive` 持有 UI 状态
  - 复用 `usePublishSettingStore()` 读写持久化配置

不新增独立 Pinia Store 的原因：

- V2 状态本质是局部 UI 状态，生命周期跟随 V2Host 的打开/关闭
- Pinia Store 是全局单例，关闭面板后状态残留不符合预期
- Composable 模式更贴合 Vue 3 组合式 API 的语义，每个 V2 实例拥有独立状态

### 7.5 业务能力复用

继续复用：

- 平台配置存储
- 平台列表解析
- 发布逻辑
- 文档上下文读取
- 平台元数据
- 宿主原生 UI 与 `siyuan-style.css`

注意：

- 复用业务逻辑是允许的
- 继续把旧 SPA 页面整体塞进 iframe，不应成为 V2 的长期实现方式
- 对 UI 而言，优先复用宿主原生样式和交互，不优先新造组件库

### 7.6 平台表单桥接策略

现有平台设置是大量专用组件，不适合首轮统一重写。

因此采用两段式策略：

1. 第一阶段：V2 设置容器桥接旧表单能力
2. 第二阶段：逐步提炼公共字段，演进到 schema 驱动

这里的“桥接”优先指：

- 复用配置解析
- 复用业务提交逻辑
- 复用已有字段定义

而不是长期保留对旧 iframe 页面本身的依赖。

### 7.7 国际化收敛策略

当前仓库里实际存在两套国际化来源：

- `src/locales/*`：历史 SPA 运行时，调用习惯是 `t("key")`
- `siyuan/i18n/*`：插件宿主运行时，调用入口是 `pluginInstance.i18n`

当前 V2 Host 已经把 `pluginInstance.i18n` 注入到 V2 的 Vue i18n 运行时中。这意味着：

- V2 实际已经具备“单一宿主 i18n 入口”
- 当前缺失的不是入口，而是 key 资产和迁移策略
- 若继续让 V2 直接依赖 `src/locales/*`，就会再次长出第二套 locale 真相源

#### 7.7.1 目标

- V2 发布态、设置态和后续 V2 原生组件只认一套 i18n 真相源：`siyuan/i18n/*`
- V2 所需的历史 SPA key 必须先镜像到 `siyuan/i18n/*`，再迁移调用方式
- 在 V1 未废弃前，桥接组件不得因为 i18n 改造而失效

#### 7.7.2 迁移规则

1. 对 V2 原生界面：
   - 新增文案统一写入 `siyuan/i18n/*`
   - 禁止再为 V2 原生界面新增独立 `src/locales/*` key

2. 对历史 SPA key：
   - 先盘点当前发布态、设置态、桥接态实际使用的 key
   - 把这些 key 以兼容形式镜像到 `siyuan/i18n/*`
   - 在 key 尚未镜像完成前，不应开始大范围改写调用

3. 对桥接组件：
   - 若组件仍被 V1 直接使用，不得在原地强改 i18n 调用方式
   - V1/V2 共存期应优先使用兼容层、镜像 key 或拆分 wrapper，而不是直接破坏共享组件
   - 只有当 V1 已退役，或者桥接组件已经从 V1 调用链中脱离，才允许彻底改成 V2-only 的 i18n 方案

#### 7.7.3 兼容策略

V1 共存期建议采用以下顺序：

1. 先把 V2 当前界面所需 key 收拢到 `siyuan/i18n/*`
2. 再为 V2 建立统一 i18n gateway / composable
3. V2 原生组件优先切换到宿主 i18n
4. 共享桥接组件在 V1 退役前保留兼容调用或增加 wrapper

这条策略的核心是：

> 先收拢 key，再迁移调用；先保证 V1 不坏，再推进桥接组件重构。

#### 7.7.4 `useV2I18n` 双层 fallback 机制

实际实现采用 `useV2I18n` composable，提供三层解析：

1. **Vue i18n locale messages**：从 `createI18n({ messages })` 中解析 `t("v2.xxx")`
2. **宿主 i18n fallback**：通过 `inject(V2_I18N_FALLBACK_KEY)` 调用 `pluginInstance.i18n` 解析
3. **key 回退**：前两层均缺失时，直接返回 key 本身

该机制保证：

- V2 新增文案统一写入 `siyuan/i18n/*`，通过 `pluginInstance.i18n` 作为 fallback 生效
- 不需要在 `src/locales/*` 中维护第二套 key
- `V2Host` 在创建 Vue 应用时注入 `fallbackResolve`，桥接组件即使使用旧 key 也能通过 fallback 正常显示

## 8. 全生命周期路线图

### Milestone 0: 入口与治理基座

#### 目标

建立能运行、能回退、能扩展的 V2 基座。

#### 范围

- 统一入口定义
- `useV2UI` 开关
- `V2Host`
- 单一配置源
- 回退机制
- 明确 V2 主路径只允许真实 DOM 挂载
- V2 源码以常规组件形式集成到现有构建链，不新增独立构建入口

#### 输出

- 稳定可开的 V2 入口
- 失败自动回退
- 最小可运行 `V2App`

#### 退出条件

- 开关关：旧 UI
- 开关开：V2 Host
- Host 失败：旧 UI
- 关键偏好读取路径唯一

### Milestone 1: 样式系统与工作壳骨架

#### 目标

建立 V2 样式基础，为后续主界面和设置展开态提供统一视觉骨架。

#### 范围

- `.syp-v2` 命名空间
- 设计令牌
- 基础排版、按钮、卡片
- `UnifiedWorkspaceShell` 骨架样式

#### 输出

- 样式统一入口
- 品牌区 / 导航区 / 内容区 / 详情区骨架

#### 退出条件

- V2 样式不污染旧 UI
- 统一壳骨架可渲染
- 样式不再双轨

### Milestone 2: 快速发布主界面

#### 目标

交付主界面态，让 V2 第一屏真正成为快速发布界面。

#### 范围

- 当前文档信息展示
- 已配置平台列表
- 空状态
- 设置展开入口

#### 输出

- 主界面态可展示真实平台
- 用户可从主界面切到设置展开态

#### 退出条件

- 已配置态和空状态都可工作
- 不需要进入设置即可完成平台选择动作准备

### Milestone 3: 发布动作闭环

#### 目标

打通主界面态的发布闭环。

#### 范围

- 单平台发布
- 发布中状态
- 发布成功/失败反馈
- 重试

#### 输出

- V2 主界面态完成真实发布

#### 退出条件

- 用户可以在 V2 里完成当前文档发布
- 失败不阻断后续操作

### Milestone 4: 设置展开态第一阶段

#### 目标

在统一工作壳内完成设置展开态的第一阶段能力。

#### 范围

- 账号列表
- 平台选择
- 图床设置内容区
- 偏好设置内容区
- 至少一类平台配置桥接
- 开始把高频设置能力从 iframe SPA 中抽离

#### 输出

- 统一工作壳展开态可用
- 设置分类可切换
- 部分桥接链路可跑通

#### 退出条件

- 新增账号流程可用
- 图床设置可展示与保存
- 偏好设置可展示与保存

### Milestone 5: 设置展开态第二阶段

#### 目标

扩展设置展开态，逐步替换高频旧设置能力。

#### 范围

- 更多平台配置桥接
- 设置态交互打磨
- 图床和偏好高频能力稳定化
- 发布态、设置态与桥接态国际化收敛

#### 输出

- 高频设置路径优先完成迁移
- V2 当前高频界面完成宿主 i18n 收拢

#### 退出条件

- 大多数用户高频设置场景不再依赖旧设置页
- V2 当前高频界面不再依赖第二套独立 locale 真相源

### Milestone 6: 收敛与稳定发布

#### 目标

完成 V2 与旧 UI 的长期共存策略、收敛策略和稳定发布策略。

#### 范围

- 统计仍依赖旧 UI 的能力
- 判断哪些旧入口可以下沉或隐藏
- 准备后续废弃计划
- 统计仍依赖 iframe SPA 的能力
- 制定 iframe 退役清单

#### 输出

- V2 稳定发布策略
- 旧 UI 收敛清单

#### 退出条件

- 至少一个稳定版本周期内无阻断性问题
- 回退路径仍然可用
- 旧 UI 是否废弃有明确判据

## 8.1 SPA 代码退役标准

### 8.1.1 术语定义

- **SPA 代码**：指基于 `iframeDialog.ts` + Vue Router + `src/pages/*` + `src/components/set/*` 的历史页面体系，包含路由配置、页面组件、对话框逻辑和旧设置表单
- **V2 架构**：指基于 `V2Host`（思源原生 Menu 挂载）+ `createV2VueApp`（Vue 工厂）+ `UnifiedWorkspaceShell`（统一工作壳）+ `src/components/v2/*` 的原生 DOM 挂载体系
- **桥接组件**：指 `V2PlatformConfigBridge` 通过 `bridgeRegistry.ts` 复用的旧平台配置组件（如 `WordpressSetting.vue`、`CnblogsSetting.vue` 等），这些组件被 V1 SPA 和 V2 同时引用
- **功能等价性**：V2 实现覆盖对应 SPA 页面的全部用户可见功能和交互路径，用户可以在不感知差异的情况下完成相同任务

### 8.1.2 SPA 代码分类

按当前仓库实际状况，SPA 代码分为三类：

| 分类 | 说明 | 当前实例 |
|------|------|----------|
| **可替代核心功能** | V2 已实现等价功能，可直接替换 | 快速发布、账号列表、平台选择、图床设置、偏好设置 |
| **桥接依赖功能** | V2 已桥接，但底层仍依赖旧组件 | 26 个平台的配置表单（`V2PlatformConfigBridge`） |
| **未迁移功能** | V2 尚无等价实现，必须保留 | 常规发布、批量分发、AI 聊天、文章管理/仪表盘、测试页面、关于页面 |

### 8.1.3 安全移除条件

某段 SPA 代码被认定为**可以安全移除**，必须同时满足以下全部条件：

1. **功能等价性**：V2 实现覆盖该 SPA 页面的全部用户可见功能和交互路径，不存在 V2 缺失但 SPA 存在的操作
2. **数据兼容性**：V2 实现读写与 SPA 页面相同的数据格式和存储位置（`usePublishSettingStore` / `PreferenceConfigManager`），不引入新的持久化格式
3. **调用链替换**：所有调用该 SPA 页面的入口（`topbar.ts`、`widgetInvoke.ts`、文档菜单）已切换到 V2 路径（`V2Host.show()`）
4. **稳定性验证**：该 V2 功能已在至少一个完整里程碑周期内稳定运行，无阻断性 bug
5. **回退可行性**：即使移除该 SPA 代码，关闭 `useV2UI` 开关后，用户仍能通过其他 SPA 路径或降级方案完成相同任务

### 8.1.4 必须移除条件

某段 SPA 代码满足以下**任一条件**时，即触发移除：

1. **维护负担**：该 SPA 页面依赖的第三方库版本已停止维护，存在已知安全风险且无法升级
2. **技术债务**：该 SPA 页面阻塞了 V2 的进一步演进（如全局样式冲突、Vue Router 与 V2 状态管理冲突、构建产物体积膨胀）
3. **双系统成本**：同一功能在 SPA 和 V2 中并行维护，维护成本已超过保留价值，且 V2 路径已稳定
4. **用户覆盖率**：该功能的 V2 实现已达到 100% 用户可用状态超过一个稳定版本周期，旧路径无活跃使用

### 8.1.5 仍需保留条件

某段 SPA 代码满足以下**任一条件**时，必须保留作为兼容层：

1. **功能缺失**：V2 尚未实现该 SPA 页面的等价功能（如常规发布、批量分发、AI 聊天）
2. **桥接未完成**：该 SPA 承载的功能被 V2 部分桥接，但桥接范围未覆盖全部子类型（如平台配置桥接目前覆盖 26 个子类型，若新增平台尚未桥接）
3. **回退依赖**：V2 功能开关（`useV2UI`）关闭后，用户必须能完整回退到旧体验，此时被回退依赖的 SPA 页面不可移除
4. **外部依赖**：其他尚未迁移的 SPA 页面通过路由跳转、组件引用或共享状态依赖该页面
5. **测试依赖**：自动化测试或手工 smoke 测试仍依赖该 SPA 路径验证旧功能

### 8.1.6 转换边界定义

从 iframe SPA 到原生 DOM 挂载的转换边界按以下四层划分：

**边界 1 —— 入口层边界**
- SPA 侧：`widgetInvoke.ts` 中的 `showPage()` / `showTab()` 调用，`topbar.ts` 中的旧菜单调用
- V2 侧：`V2Host.show()` 调用
- **判定**：当某功能的 `widgetInvoke` / `topbar` 调用已被替换为 `V2Host.show()` 时，该功能的入口边界已转换

**边界 2 —— 视图层边界**
- SPA 侧：`src/pages/*.vue` 页面组件 + `routeConfig.ts` 中的 Vue Router 路由
- V2 侧：`src/components/v2/*.vue` 组件 + `V2App.vue` 中的条件渲染（`v-if` / `v-else-if`）
- **判定**：当某页面组件在 V2 中有等价实现且不再被 iframe 加载时，视图边界已转换

**边界 3 —— 配置层边界**
- SPA 侧：`src/stores/*` 中的旧配置读取逻辑（部分页面直接操作 `window.localStorage`）
- V2 侧：`usePublishSettingStore` / `PreferenceConfigManager` 统一读写
- **判定**：当某功能的所有配置读写都通过统一 Store 完成，不再存在双读双写时，配置边界已转换

**边界 4 —— 国际化边界**
- SPA 侧：`src/locales/*` 中的 key（`t("key")` 调用）
- V2 侧：`siyuan/i18n/*` 中的 key + `useV2I18n` fallback 机制
- **判定**：当某功能的所有用户可见文案都已镜像到 `siyuan/i18n/*`，且 V2 不再新增 `src/locales/*` key 时，国际化边界已转换

### 8.1.7 功能完全迁移判定标准

判断某个功能已**完全**从 SPA 迁移到 V2 架构，必须通过以下检查清单：

- [ ] 该功能的所有用户交互路径在 V2 中已实现，无操作断点
- [ ] 该功能的 V2 实现不依赖 iframe 加载任何 SPA 页面（包括嵌套 iframe）
- [ ] 该功能的数据读写格式与 SPA 版本一致，共享同一持久化存储（`PreferenceConfigManager` / `usePublishSettingStore`）
- [ ] 该功能的入口调用（`topbar.ts` / `widgetInvoke.ts` / 文档菜单）已切换到 `V2Host`
- [ ] 该功能的国际化 key 已收拢到 `siyuan/i18n/*`，且 `src/locales/*` 中无新增依赖
- [ ] 该功能在至少一个发布周期内未收到 V2 路径的阻断性 bug 反馈
- [ ] 该功能的 V2 实现有明确的回退策略（关闭 `useV2UI` 后仍可回退到 SPA 或其他可用路径）
- [ ] 该功能对应的 SPA 路由在 `routeConfig.ts` 中已被标记为废弃或已移除

### 8.1.8 进度衡量指标

衡量 SPA 代码退役进度的量化指标体系：

**指标 1 —— 页面退役率**
```
页面退役率 = 已移除的 SPA 页面数 / 总 SPA 页面数
```
- 当前总 SPA 页面：按 `routeConfig.ts` 统计约 20 个路由对应页面/组件
- 目标：Phase 5 达到 100%

**指标 2 —— 入口切换率**
```
入口切换率 = 已切换到 V2Host 的功能入口数 / 总功能入口数
```
- 当前总功能入口：按 `topbar.ts` + `widgetInvoke.ts` 统计约 10 个
- 当前状态：顶栏快速发布、发布设置已切换（2/10 = 20%）
- 目标：Phase 3 达到 80%+

**指标 3 —— 路由收敛率**
```
路由收敛率 = 已废弃的 routeConfig 路由数 / 总路由数
```
- 当前总路由数：`routeConfig.ts` 中约 20 条路由
- 目标：Phase 5 达到 100%

**指标 4 —— i18n 收敛率**
```
i18n 收敛率 = 已镜像到 siyuan/i18n 的 V2 key 数 / V2 总 key 数
```
- 当前状态：V2 新增 key 已 100% 在 `siyuan/i18n/*` 中
- 目标：始终保持 100%，禁止新增 `src/locales/*` key

**指标 5 —— 桥接覆盖率**
```
桥接覆盖率 = V2 已桥接的平台子类型数 / 总平台子类型数
```
- 当前状态：26 个桥接子类型 / 总平台子类型（约 30+）
- 目标：Phase 4 逐步提升到 100%，然后启动桥接替换（用 V2 原生表单替换桥接）

**指标 6 —— 版本稳定性指标**
```
版本稳定性 = 连续无阻断性 bug 的版本数
```
- 判定标准：连续 2 个次要版本或 1 个主要版本无 V2 阻断性 bug
- 目标：每个里程碑退出前必须达到

### 8.1.9 实施路线图

**Phase 1 —— 稳定期（当前，Milestone 4 阶段）**
- 目标：V2 核心功能（快速发布 + 设置展开态）稳定运行
- SPA 状态：全部保留，无移除
- 判定条件：Milestone 4 退出条件达成（新增账号流程可用、图床设置可展示与保存、偏好设置可展示与保存）

**Phase 2 —— 核心退役（Milestone 5-6）**
- 目标：将已有 V2 等价实现的功能从 `widgetInvoke.ts` 中移除 iframe 调用
- SPA 状态：发布设置（`/setting/publish`）路由可标记为废弃；账号设置、图床设置、偏好设置的旧 iframe 入口可隐藏
- 判定条件：上述功能的 V2 路径连续两个版本无阻断性 bug，且 `useV2UI` 默认开启后无用户投诉

**Phase 3 —— 全面退役（Milestone 6 之后）**
- 目标：常规发布、批量分发、AI 聊天、文章管理等剩余功能迁移到 V2
- SPA 状态：仅保留测试页面、关于页面等低频或开发辅助功能
- 判定条件：上述功能在 V2 中实现并通过功能等价性检查清单

**Phase 4 —— 桥接替换（长期）**
- 目标：用 V2 原生平台配置表单逐步替换 `V2PlatformConfigBridge` 对旧组件的依赖
- SPA 状态：可逐步移除 `src/components/set/publish/singleplatform/*` 中已被替换的组件
- 判定条件：某平台子类型的 V2 原生表单已实现、通过测试，且该平台在 V1 中已无直接引用

**Phase 5 —— 终局清理（V2 稳定发布后）**
- 目标：移除 `iframeDialog.ts`、Vue Router（`routeConfig.ts`）、所有 SPA 页面组件（`src/pages/*`）
- SPA 状态：完全退役
- 判定条件：`useV2UI` 开关默认值设为 `true`，旧入口全部移除，连续一个主版本周期（如 3.0.0 到 4.0.0）无回退需求

## 9. 渐进迁移策略

### 9.1 共存策略

- 默认允许旧 UI 与 V2 共存
- V2 通过开关启用
- 旧入口不立即删除
- iframe 路径仅作为历史兼容路径，不能继续承载 V2 新能力

### 9.2 桥接策略

- 能桥接就桥接
- 不能桥接的先保留旧页
- 不用“一定全新重写”绑死后续阶段
- 但桥接优先到真实 DOM 组件层，不优先到 iframe 页面层
- V1 共存期不得通过桥接改造破坏共享组件的 i18n 可用性

### 9.3 国际化策略

- V2 原生界面统一走 `siyuan/i18n/*`
- 旧 SPA key 先镜像到宿主 i18n，再迁移调用方式
- V1 未退役前，不允许直接改写共享桥接组件导致 V1 不可用

### 9.4 回退策略

以下情况可直接回退到上一个稳定点：

- 入口不稳定
- 配置双读
- V2 影响旧流程
- 某一阶段的实现破坏前置阶段验收

## 10. 验证策略

### 10.1 每个里程碑都必须具备

- 手工 smoke 清单
- 至少一项自动化或半自动验证
- 回退验证

### 10.2 重点验证对象

- 顶栏入口
- 文档快捷入口
- 偏好开关
- 主界面平台展示
- 发布动作
- 账号设置
- 图床设置
- 偏好设置

## 11. 风险

### 风险 1：仍然按局部思维推进

结果会是：

- 某个里程碑优化局部
- 后续阶段再推翻
- 总体成本升高

### 风险 2：过早重写所有平台表单

结果会是：

- 开发周期极长
- 频繁返工
- 主路径迟迟不能交付

### 风险 3：缺少生命周期管理

结果会是：

- 每一阶段都像一个临时 patch
- 维护者无法判断接下来该做什么

### 风险 4：继续维护 iframe 双系统

结果会是：

- DOM 版本和 iframe 版本长期重复演进
- 存储和入口逻辑持续分叉
- 维护成本继续上升

## 12. 决策

- V2 必须按全生命周期方案推进
- 仍然保留强门禁：前一里程碑未通过，禁止进入下一里程碑实现
- 优先交付主路径，再逐步吸收设置能力
- V2 的长期方向必须是 DOM-only，iframe 仅作为过渡兼容层
- 允许慢，但不允许乱
