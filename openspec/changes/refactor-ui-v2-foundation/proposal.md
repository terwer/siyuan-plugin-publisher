# Change: UI V2 全生命周期重构方案与渐进迁移

## Why

当前 UI V2 讨论已经从"做一个实验入口"升级为"重构整个发布工具前端工作流"。如果继续只做局部基建，会带来五个直接问题：

1. 后续阶段没有统一目标，容易在不同提交中反复改变方向
2. 里程碑之间缺少前后约束，导致上一阶段未稳定就开始下一阶段
3. 快速发布、账号设置、图床设置、偏好设置、平台配置桥接、批量分发、文章管理仪表盘、详细发布模式之间没有全局设计，会在后期产生更高维护成本
4. V2 快速发布目前只支持默认配置，不支持自定义摘要、标签、分类等高级字段，缺少详细设置入口
5. 国际化仍分散在 `src/locales/*` 与 `siyuan/i18n/*` 两套体系中，V2 若不先定义统一策略，会在桥接期同时伤到 V1 与 V2

因此本次变更不再把 V2 视为一个单点基础设施任务，而是视为一个完整的产品与工程迁移计划。它需要覆盖：

- 统一工作壳模型
- 从快速发布到完整设置的同壳渐进展开
- 快速发布支持详细模式（摘要、标签、分类等高级字段）
- 批量分发多平台发布能力（先桥接，后评估）
- 文章管理仪表盘能力（先桥接，后评估）
- 旧 UI 与 V2 的长期共存与逐步替换
- 从 `Milestone 0` 到最终稳定发布的完整生命周期
- 从 `iframe + SPA` 历史结构向真实 DOM 挂载的长期迁移
- 从双轨国际化到单一宿主国际化的渐进收敛

## What Changes

- 将现有 V2 提案从"基座治理"升级为"全生命周期迁移方案"
- 将"优先使用思源原生 UI 与 `siyuan-style.css` 体系"设为 V2 最高优先级设计规则
- 明确 V2 的统一工作壳模型，而不是分裂成多套页面心智
- 定义 V2 的完整范围：快速发布、详细发布模式、账号设置、图床设置、偏好设置、平台配置桥接、批量分发桥接、文章管理仪表盘桥接、旧 UI 渐进替换
- 明确 V2 的总体方向是逐步淘汰 iframe 页面承载模式，只保留真实 DOM 挂载
- 新增 M8 详细发布模式整合里程碑，在快速发布页面添加详细设置入口
- 明确批量分发、文章管理仪表盘和详细发布模式采用渐进策略：M5/M7/M8 先桥接现有实现，M6/M9 评估是否需要原生重写
- 明确 V2 国际化以 `siyuan/i18n/*` 为单一真相源，不再为 V2 维持第二套独立 locale 包
- 明确旧 SPA key 的镜像与调用迁移策略，要求先把 V2 所需 key 收拢到思源 i18n，再迁移调用方式
- 明确桥接组件在 V1 退役前必须保持兼容，禁止直接改写共享组件导致 V1 文案失效
- 定义完整里程碑 `M0-M9`，并为每个里程碑提供目标、输入、输出、退出条件和回退规则
- 新增 M5 批量分发桥接、M7 文章管理仪表盘桥接、M8 详细发布模式整合里程碑，原 M5-M6 后移为 M6、M9
- 明确配置兼容、桥接策略、验证策略、发布策略和风险控制
- 补充 SPA 代码退役的完整标准体系：安全移除条件、必须移除条件、兼容层保留条件、转换边界定义、功能迁移判定清单、进度衡量指标、五阶段实施路线图
- 明确 V2 仅服务于思源插件运行时（PC Electron + Docker 浏览器端），Chrome Extension / Nginx / Vercel / Widget 继续使用 SPA
- 明确 SPA 退役范围限定为插件运行时路径（隐藏 iframe 入口、替换为 `V2Host.show()` 调用），而非从代码库中删除 SPA 代码
- 保留严格门禁：上一个里程碑未通过，不得开始下一个里程碑实现

## Impact

- 影响范围首先是文档、设计与实施约束，但它将指导后续全部 V2 实现
- 后续将影响 V2 入口、V2 Host、统一工作壳、快速发布主界面、详细发布模式、设置展开态、图床设置、偏好设置、平台配置桥接、批量分发桥接和文章管理仪表盘桥接
- 后续将约束对 `siyuan/topbar.ts`、`siyuan/invoke/widgetInvoke.ts`、V2 组件目录、V2 状态层和配置层的修改方式
- 后续将约束 `src/locales/*`、`siyuan/i18n/*`、V2 i18n gateway 与桥接组件的演进方式
- 后续将约束 iframe 对话框、SPA 路由页和旧桥接页的保留与退役方式
- 后续将约束 V2 的组件和样式实现方式，优先走思源原生 UI
- 本变更不应修改现有平台配置格式、适配器协议和发布逻辑契约
- 本变更允许在必要时回滚当前未稳定的实验性 V2 运行时代码，但不允许无方案地继续向后堆功能
- 批量分发、文章管理仪表盘和详细发布模式在 M5/M7/M8 阶段采用桥接策略，不立即重写，在 M6/M9 收敛阶段评估是否需要原生实现
