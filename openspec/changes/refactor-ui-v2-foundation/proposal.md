# Change: UI V2 全生命周期重构方案与渐进迁移

## Why

当前 UI V2 讨论已经从“做一个实验入口”升级为“重构整个发布工具前端工作流”。如果继续只做局部基建，会带来三个直接问题：

1. 后续阶段没有统一目标，容易在不同提交中反复改变方向
2. 里程碑之间缺少前后约束，导致上一阶段未稳定就开始下一阶段
3. 快速发布、账号设置、图床设置、偏好设置、平台配置桥接之间没有全局设计，会在后期产生更高维护成本

因此本次变更不再把 V2 视为一个单点基础设施任务，而是视为一个完整的产品与工程迁移计划。它需要覆盖：

- 统一工作壳模型
- 从快速发布到完整设置的同壳渐进展开
- 旧 UI 与 V2 的长期共存与逐步替换
- 从 `Milestone 0` 到最终稳定发布的完整生命周期
- 从 `iframe + SPA` 历史结构向真实 DOM 挂载的长期迁移

## What Changes

- 将现有 V2 提案从“基座治理”升级为“全生命周期迁移方案”
- 将“优先使用思源原生 UI 与 `siyuan-style.css` 体系”设为 V2 最高优先级设计规则
- 明确 V2 的统一工作壳模型，而不是分裂成多套页面心智
- 定义 V2 的完整范围：快速发布、账号设置、图床设置、偏好设置、平台配置桥接、旧 UI 渐进替换
- 明确 V2 的总体方向是逐步淘汰 iframe 页面承载模式，只保留真实 DOM 挂载
- 定义完整里程碑 `M0-M6`，并为每个里程碑提供目标、输入、输出、退出条件和回退规则
- 明确配置兼容、桥接策略、验证策略、发布策略和风险控制
- 保留严格门禁：上一个里程碑未通过，不得开始下一个里程碑实现

## Impact

- 影响范围首先是文档、设计与实施约束，但它将指导后续全部 V2 实现
- 后续将影响 V2 入口、V2 Host、统一工作壳、快速发布主界面、设置展开态、图床设置、偏好设置和平台配置桥接
- 后续将约束对 `siyuan/topbar.ts`、`siyuan/invoke/widgetInvoke.ts`、V2 组件目录、V2 状态层和配置读取层的修改方式
- 后续将约束 iframe 对话框、SPA 路由页和旧桥接页的保留与退役方式
- 后续将约束 V2 的组件和样式实现方式，优先走思源原生 UI
- 本变更不应修改现有平台配置格式、适配器协议和发布逻辑契约
- 本变更允许在必要时回滚当前未稳定的实验性 V2 运行时代码，但不允许无方案地继续向后堆功能
