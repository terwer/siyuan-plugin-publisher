# FS图片上传根因修复：delegate newMediaObject 重定向

## 根因

`LocalSystemApiAdaptor.preEditPost()` 委派给 GitHub 系子 adaptor（Hexo/Hugo/VuePress 等）的完整 `preEditPost()` 链路。子 adaptor 内部执行统一图片处理时，`handlePictures()` 调用的是子 adaptor 自己实例上的 `newMediaObject()`，图片最终走到了 GitHub API。

**不是 `newMediaObject()` 实现错了，而是调错了实例。**

## 修复核心思路

保留子 adaptor 的完整 `preEditPost()` 链路，但在委派前将子 adaptor 的 `newMediaObject()` 重定向到 `LocalSystemApiAdaptor.newMediaObject()`。

```typescript
const delegate = new HexoApiAdaptor(this.appInstance, localFsCfg as any)
delegate.newMediaObject = this.newMediaObject.bind(this)
updatedPost = await delegate.preEditPost(post, id, publishCfg)
```

这样：
- 统一图片处理链（发现→上传→urlMap→链接替换）照常运行
- 平台的 markdown/yaml/`<!-- more -->` 收尾逻辑照常运行
- 只有"媒体最终存储后端"从 GitHub 切到了本地文件系统

---

## 为什么这条路直接命中根因

`handlePictures()` Bundled 分支调用 `this.api.newMediaObject(mediaObject)`。`this.api` 是 `BaseExtendApi` 构造时存入的 delegate 实例引用。在 delegate 实例上替换 `newMediaObject` 后，`this.api.newMediaObject()` 命中替换后的方法，即 `LocalSystemApiAdaptor.newMediaObject()` → 写本地文件系统。

整条图片处理链不变，链接替换逻辑不变，平台收尾逻辑不变，只有媒体存储后端被替换。

---

## 为什么其他方案不如这个

| 方案 | 否定原因 |
|------|----------|
| 改 base 新增公共方法 | 影响面大，公共接口膨胀，不适合为一个特例付出全局复杂度 |
| 不委派子adaptor，直接调 `super.preEditPost()` | 丢失子adaptor的平台收尾逻辑（如 Vuepress `<!-- more -->`），需另起一套 formatter 补 |
| 新增第二套 markdown 图片替换逻辑 | 仓库已有统一链路，严禁复制 |
| 临时设 `picbedService=None` 再补图片 | 需额外流程控制，需恢复现场，且"补图片"环节仍需解决"用什么上传后端"问题 |
| 用 publishCfg 加流程标记 | 把链路切得越来越复杂，不直接命中根因 |

---

## 修复方案

### 改动1: LocalSystemApiAdaptor — 统一创建 delegate + 重定向 newMediaObject

**文件**: `src/adaptors/fs/LocalSystem/LocalSystemApiAdaptor.ts`

#### 1.1 新增 `createDelegateAdaptor()` private 方法

统一创建子 adaptor 并重定向 `newMediaObject()`，避免在每个 case 里散落式重复 patch 逻辑：

```typescript
/**
 * 创建委派 adaptor 并将媒体上传重定向到本地文件系统
 * 
 * 子 adaptor 的完整 preEditPost() 链路照常运行（包括统一图片处理、YAML 格式化、平台收尾逻辑），
 * 但 handlePictures() 中调用的 newMediaObject() 会被重定向到 LocalSystemApiAdaptor 自身的方法，
 * 使图片最终落盘到本地文件系统而非 GitHub。
 */
private createDelegateAdaptor(fsYamlType: FsYamlType): BaseBlogApi {
  const localFsCfg = this.cfg as LocalSystemConfig
  let delegate: BaseBlogApi

  switch (fsYamlType) {
    case FsYamlType.Hexo:
      delegate = new HexoApiAdaptor(this.appInstance, localFsCfg as any)
      break
    case FsYamlType.Hugo:
      delegate = new HugoApiAdaptor(this.appInstance, localFsCfg as any)
      break
    case FsYamlType.Jekyll:
      delegate = new JekyllApiAdaptor(this.appInstance, localFsCfg as any)
      break
    case FsYamlType.Vuepress:
      delegate = new VuepressApiAdaptor(this.appInstance, localFsCfg as any)
      break
    case FsYamlType.Vuepress2:
      delegate = new Vuepress2ApiAdaptor(this.appInstance, localFsCfg as any)
      break
    case FsYamlType.Vitepress:
      delegate = new VitepressApiAdaptor(this.appInstance, localFsCfg as any)
      break
    case FsYamlType.Quartz:
      delegate = new QuartzApiAdaptor(this.appInstance, localFsCfg as any)
      break
    case FsYamlType.Astro:
      delegate = new AstroApiAdaptor(this.appInstance, localFsCfg as any)
      break
    default:
      throw new Error(`不支持的 fsYamlType: ${fsYamlType}`)
  }

  // 核心修复：将子 adaptor 的媒体上传重定向到本地文件系统
  // handlePictures() 中 this.api.newMediaObject() 将命中此替换方法
  delegate.newMediaObject = this.newMediaObject.bind(this)

  return delegate
}
```

需新增 import：
```typescript
import { AstroApiAdaptor } from "~/src/adaptors/api/astro/astroApiAdaptor.ts"
import { BaseBlogApi } from "~/src/adaptors/api/base/baseBlogApi.ts"
```

#### 1.2 修改 preEditPost() — 使用统一委派入口

```typescript
public override async preEditPost(post: Post, id?: string, publishCfg?: any): Promise<Post> {
  const localFsCfg = this.cfg as LocalSystemConfig

  if (localFsCfg.storePath.includes(CATE_AUTO_NAME)) {
    post.cate_slugs = [CATE_AUTO_NAME]
  }

  let updatedPost: Post

  // 判断是否需要委派给子 adaptor
  const knownFsYamlTypes = [
    FsYamlType.Hexo, FsYamlType.Hugo, FsYamlType.Jekyll,
    FsYamlType.Vuepress, FsYamlType.Vuepress2, FsYamlType.Vitepress,
    FsYamlType.Quartz, FsYamlType.Astro,
  ]

  if (knownFsYamlTypes.includes(localFsCfg.fsYamlType)) {
    // 委派给子 adaptor（完整链路），但图片上传重定向到本地文件系统
    const delegate = this.createDelegateAdaptor(localFsCfg.fsYamlType)
    updatedPost = await delegate.preEditPost(post, id, publishCfg)
  } else {
    // 默认走 LocalSystem 自身的 base 链路
    updatedPost = await super.preEditPost(post, id, publishCfg)
  }

  if (updatedPost?.cate_slugs.length > 0) {
    localFsCfg.realStorePath = localFsCfg.storePath.replace(CATE_AUTO_NAME, updatedPost.cate_slugs[0])
  }

  return updatedPost
}
```

**为什么不会二次处理**：只调用了一次 `delegate.preEditPost()`，完整链路执行一次。不存在第二次调用。

**为什么不需要临时切换配置**：不需要设 `picbedService=None`。子 adaptor 的 `handlePictures()` 正常进入 Bundled 分支，正常调用 `newMediaObject()`，只是命中了重定向后的本地实现。

**为什么不需要恢复现场**：没有修改任何运行时配置，没有临时状态，不存在脏状态风险。

### realStorePath / 自动分类路径时序修正

#### 风险

当 `storePath` 包含 `CATE_AUTO_NAME`（自动分类）时：

1. `preEditPost()` 开头设 `post.cate_slugs = [CATE_AUTO_NAME]`
2. delegate 内部 `handleCategories()` 将 `post.cate_slugs` 解析为真实分类路径
3. delegate 内部 `handlePictures()` 调用 `LocalSystemApiAdaptor.newMediaObject()`
4. `newMediaObject()` 检查 `localFsCfg.realStorePath`——但此时 `realStorePath` 尚未设置（它在 `preEditPost()` 尾部才赋值）
5. 回退到 `localFsCfg.storePath`，其中仍含 `CATE_AUTO_NAME`，图片落到错误目录

#### 修正策略

在 `LocalSystemApiAdaptor.newMediaObject()` 中，当 `realStorePath` 未设置但 `storePath` 含 `CATE_AUTO_NAME` 时，从 `mediaObject.post.cate_slugs` 动态解析真实路径。

`mediaObject.post` 由 `handlePictures()` 在 L512-513 设置：
```typescript
const mediaObject = new MediaObject(image.name, base64Info.mimeType, bits)
mediaObject.post = post  // ← 此时 post.cate_slugs 已被 handleCategories() 处理过
```

`handleCategories()` 在 `handlePictures()` 之前执行（preEditPost 7步链），所以 `post.cate_slugs` 在图片上传时已是真实分类路径。

修正代码（在 `newMediaObject()` 的路径解析部分，与跨文档残留修正的 B 策略一致）：

```typescript
// 存储基础路径
let storePath = localFsCfg.storePath
if (storePath.includes(CATE_AUTO_NAME) && mediaObject.post?.cate_slugs?.length > 0) {
  // 自动分类场景：优先使用当前文档的分类路径，不信任历史缓存
  storePath = storePath.replace(CATE_AUTO_NAME, mediaObject.post.cate_slugs[0])
} else if (!StrUtil.isEmptyString(localFsCfg.realStorePath)) {
  // 非自动分类场景，或 post.cate_slugs 不可用时，使用 realStorePath
  storePath = localFsCfg.realStorePath
}
```

#### 为什么这个策略成立

- **时序正确**：`handleCategories()` → `handlePictures()` → `newMediaObject()`，post.cate_slugs 在图片上传前已被处理
- **自动分类优先**：当 `storePath` 含 `CATE_AUTO_NAME` 时，优先从当前 post 推导路径，不信任历史缓存（与跨文档残留修正 B 策略一致）
- **非自动分类不受影响**：`storePath` 不含 `CATE_AUTO_NAME` 时，第一个 `if` 不命中，走 `realStorePath` 分支
- **realStorePath 已设置时不受影响**：`newPost()`/`editPost()` 调用时，`storePath` 不含 `CATE_AUTO_NAME`，走 `realStorePath` 分支

### realStorePath 跨文档残留风险修正

#### 风险

`localFsCfg.realStorePath` 是挂在配置对象上的运行时状态，不是本次调用的局部变量。在自动分类场景下：

1. 第一篇文档（分类 a）发布后，`realStorePath` 被设为 `/path/to/a`
2. 第二篇文档（分类 b）开始发布，`preEditPost()` 开头还未清空 `realStorePath`
3. delegate 内图片上传时，`newMediaObject()` 检查 `realStorePath`——发现非空（旧值 `/path/to/a`）
4. 优先命中旧值，第二篇文档的图片被写进第一篇的目录

#### 修正策略：A + B 双重防御

**必做 A：preEditPost() 开头清空历史 realStorePath**

在 `LocalSystemApiAdaptor.preEditPost()` 开头，进入本次发布链之前，先清空上一次残留的 `realStorePath`：

```typescript
public override async preEditPost(post: Post, id?: string, publishCfg?: any): Promise<Post> {
  const localFsCfg = this.cfg as LocalSystemConfig

  // 清空上一次发布的 realStorePath 残留，防止跨文档污染
  localFsCfg.realStorePath = ""

  if (localFsCfg.storePath.includes(CATE_AUTO_NAME)) {
    post.cate_slugs = [CATE_AUTO_NAME]
  }
  // ...
}
```

目的：保证本次 delegate 内图片上传不会误用旧文档目录，让路径推导从干净状态开始。

**必做 B：newMediaObject() 自动分类场景优先使用当前 post 分类信息**

在 `newMediaObject()` 中，当 `storePath` 含 `CATE_AUTO_NAME` 时，优先使用当前 `mediaObject.post.cate_slugs` 推导真实目录，不优先信任 `realStorePath`：

```typescript
// 存储基础路径
let storePath = localFsCfg.storePath
if (storePath.includes(CATE_AUTO_NAME) && mediaObject.post?.cate_slugs?.length > 0) {
  // 自动分类场景：优先使用当前文档的分类路径，不信任历史缓存
  storePath = storePath.replace(CATE_AUTO_NAME, mediaObject.post.cate_slugs[0])
} else if (!StrUtil.isEmptyString(localFsCfg.realStorePath)) {
  // 非自动分类场景，或 post.cate_slugs 不可用时，使用 realStorePath
  storePath = localFsCfg.realStorePath
}
```

注意判断顺序的变化：**自动分类检测优先于 realStorePath**。这确保即使 A 未生效（或未来代码路径变更），自动分类场景下当前文档的分类信息始终优先。

#### 为什么这样能彻底解决问题

- **A 清空旧值**：从源头消除跨文档污染，本次发布不会读到上一篇文章的路径
- **B 优先当前 post**：防御性兜底，即使 `realStorePath` 因任何原因残留，自动分类场景下仍以当前文档的 `cate_slugs` 为准
- **不影响非自动分类场景**：`storePath` 不含 `CATE_AUTO_NAME` 时，A 的清空不影响逻辑（`realStorePath` 会在 preEditPost 尾部重新赋值），B 的第一个条件不命中，走 `realStorePath` 分支

### baseExtendApi.ts 改动范围决策：全部纳入本次

**决策：全部纳入本次——toast 删除 + 单图 try/catch + 完整错误信息。**

这些是之前的设计错误和体验问题，现在已有更好的错误归类机制（imageUploadErrors → V2 警告卡片），不应再留尾巴。

**文件**: `src/adaptors/base/baseExtendApi.ts`

##### 1. 删 default 分支 pushMsg

`picbedService=None` 是合法配置，不应弹警告 toast 误导用户。改为仅日志记录。

##### 2. 删 Bundled 分支逐图成功 toast

成功是正常行为，逐个打扰用户体验极差；图片上传错误已通过 imageUploadErrors[] 收集，在 V2 发布结果卡片统一展示。

##### 3. Bundled 分支：外层大 try/catch → 单图级别 try/catch

**行为变更**：
- 改前：整个 for 循环一个大 try/catch，一张图失败终止全部后续图片
- 改后：每张图独立 try/catch，一张失败后继续处理后续图片
- 错误信息格式：`${image.name} 同步失败(使用${picbedName}): ${message}`

新增 private 辅助方法：
```typescript
private getPicbedServiceName(cfg: BlogConfig): string {
  switch (cfg.picbedService) {
    case PicbedServiceTypeEnum.Bundled: return "平台图床"
    case PicbedServiceTypeEnum.PicGo: return "PicGo"
    case PicbedServiceTypeEnum.None: return "无"
    default: return String(cfg.picbedService || "未知")
  }
}
```
