# 发布工具 UI 重构 V2.0 执行文档

## 1. 文档定位

本文档是发布工具 V2 UI 重构的执行基线，目标不是继续罗列想法，而是把以下内容一次性定清：

- 产品目标
- 视觉与交互原则
- 技术边界
- 里程碑顺序
- 每个里程碑的准入与退出标准
- 当前实现的实际状态

本次重构必须遵守一个硬规则：

> 上一里程碑未通过验收，禁止进入下一里程碑。

本次重构还必须遵守一个最高优先级设计规则：

> 优先使用思源笔记原生 UI 与 `siyuan-style.css` 体系。只要宿主已经提供可复用的按钮、菜单、表单、布局或状态样式，就优先复用；不要重复造一套通用 UI 组件系统。

## 3. V2 重构目标

### 3.1 产品目标

V2 不是“重新做一个设置页”，而是把发布工具的主任务路径重新排布：

- 第一界面应该是“快速发布”
- 设置应该成为次级入口
- 已配置平台应该优先展示
- 平台发布操作应该比现状更短、更确定

### 3.2 工程目标

- 新旧 UI 共存
- 可通过开关回退
- 配置格式不变
- 发布逻辑尽量复用
- 迁移期间不影响旧功能
- V2 的长期目标是彻底摆脱 `iframe + SPA` 双系统
- V2 新能力只允许基于真实 DOM 挂载实现

### 3.3 非目标

以下内容不属于 V2 首轮重构目标：

- 全量重写所有平台适配器
- 一次性重写 40+ 平台配置表单
- 顺手修所有历史页面风格问题
- 在 UI 重构阶段修改配置数据结构
- 在没有验收的情况下并行推进多个里程碑
- 继续为 V2 新增任何 iframe 页面

## 4. 设计原则

### 4.0 原生优先

这是 V2 最重要的设计与实现约束，高于视觉风格偏好：

- 优先使用思源笔记原生 UI
- 优先复用思源原生类名和宿主行为
- 优先复用宿主已注入的 `siyuan-style.css`
- 非必要不重新发明按钮、输入框、菜单、面板、状态色和交互反馈

只有当以下情况同时成立时，才允许自定义样式层：

- 宿主没有提供可用能力
- 该能力对 V2 目标体验是必须的
- 局部自定义不会与宿主样式冲突

这条规则的直接收益是：

- 与宿主更一致
- 工作量显著降低
- 维护成本更低
- 主题兼容性更好

更具体地说，它的重要性体现在：

1. 让用户 `feel at home`，进入插件时不会感觉跳到另一个产品里
2. 明显减少样式体积，无需重复定义大量通用组件样式
3. 为其他扩展和后续插件间协作留下更好的兼容空间
4. 自动适配宿主主题与视觉模式
5. 在桌面、移动、浅色、深色等模式之间实现更自然的无缝衔接

因此后续所有 V2 里程碑都应优先问一个问题：

> 这部分能不能先用思源原生 UI 做出来？

### 4.1 主任务优先

用户打开发布工具，首先应该看到能直接执行的对象，而不是复杂设置结构。对本项目而言，主任务是“把当前文档发布到已配置平台”。

### 4.2 高确定性

参考 Ant Design 的“确定性”原则，V2 必须降低用户和开发者两侧的不确定性：

- 用户知道当前入口会打开什么
- 用户知道某个按钮会发生什么
- 开发者知道某个配置应该从哪里读
- 开发者知道某个里程碑完成后才能进入下一步

### 4.3 渐进展开

设置复杂，但复杂度不能在第一屏一次性出现。V2 应采用：

- 主界面：以快速发布为中心，菜单和细节默认折叠或隐藏
- 设置一级：在同一界面中展开分类导航和列表
- 设置二级：在同一工作壳中进入平台选择
- 设置三级：在同一工作壳中进入平台配置

### 4.4 单一真相源

涉及入口切换、偏好开关、平台列表、平台启用状态的关键数据，必须有唯一读取通道。禁止出现：

- 设置页写文件
- 顶栏读 localStorage
- 文档菜单读另一套缓存

### 4.5 失败可回退

V2 是渐进迁移，不是一次性替换。任意阶段只要 V2 Host 初始化失败，都必须允许回退旧 UI，不得阻断用户已有发布流程。

## 5. 视觉与交互基线

### 5.1 视觉方向

以轻量 B 端工具界面为主，不做炫技风格：

- 中性色背景与分层容器
- 蓝色作为主操作强调色
- 8px 间距体系
- 8px/12px 圆角
- 低对比辅助线，高对比操作控件

补充截图确认了以下具体视觉语义：

- 桌面设置态允许使用大面积蓝色渐变品牌区
- 品牌区用于建立识别，不承载高频操作
- 设置工作区应为高圆角白色面板，形成明确的内容焦点
- 一级导航应采用浅底卡片式高亮，而不是仅靠文字颜色变化
- 右侧内容区应使用“标题 -> 分组标题 -> 表单行”的稳定结构
- 快速发布与设置是同一个产品框架的不同展开程度，而不是两个无关页面

### 5.2 布局原则

根据现有设计图和补充截图，V2 应采用一个统一工作壳，并根据场景控制区域显隐：

#### 统一工作壳固定区域

- 品牌视觉区
- 导航区
- 主内容区
- 详情区

#### 主界面态

- 主内容区聚焦快速发布
- 导航区可以隐藏或弱化
- 详情区默认隐藏
- 品牌视觉区可以保留但不能压过主任务

#### 设置展开态

- 导航区展开
- 主内容区承接列表或表单分组
- 详情区按需要展开
- 品牌视觉区作为整体壳的一部分保留

但必须注意：

- 设计图展示的是“设置态”
- 不是 V2 的第一屏
- 第一屏必须仍是快速发布面板
- 快速发布与设置必须在同一全局框架内切换，不能割裂成两套心智模型

### 5.3 交互原则

- 列表项可读优先于装饰
- 返回路径必须清晰
- 主要动作按钮位置稳定
- 删除、关闭、取消等危险/中断动作必须与确认动作拉开层级
- 左侧导航切换应保持右侧工作区上下文稳定
- 长表单页面必须支持分组标题和统一对齐

## 6. 信息架构

### 6.1 一级入口

- 顶栏按钮
- 文档级快捷发布入口

### 6.2 V2 主视图

- `QUICK_PUBLISH`
- `SETTINGS`

### 6.2.1 统一工作壳

当用户进入完整设置态时，应在同一个工作壳中展开导航与细节，而不是切换到一套新的外壳。

### 6.3 设置子视图

- `ACCOUNT_LIST`
- `PLATFORM_SELECT`
- `PLATFORM_CONFIG`
- `IMAGE_HOSTING_LIST`
- `IMAGE_HOSTING_CONFIG`
- `PREFERENCE`

### 6.4 关键用户流

#### 快速发布

1. 用户点击顶部发布入口
2. 若启用 V2，则进入统一工作壳的主界面态
3. 读取当前文档上下文
4. 展示已启用且已授权的平台
5. 用户点击平台卡片执行发布
6. 返回发布状态

#### 添加账号

1. 用户从快速发布进入设置
2. 在同一工作壳内展开设置导航与内容区
3. 默认进入账号列表
4. 左侧导航保持可见
5. 点击“添加账号”
6. 进入平台选择
7. 选择子平台
8. 进入平台配置
9. 保存后回到账号列表
10. 返回快速发布主界面

#### 图床设置

1. 用户在统一工作壳内进入设置展开态
2. 进入“图床设置”
3. 右侧内容区显示图床表单组
4. 表单按配置组分段展示，例如 `PicGO设置`
5. 保存后保持在当前设置上下文

#### 偏好设置

1. 用户在统一工作壳内进入设置展开态
2. 进入“偏好设置”
3. 右侧内容区显示偏好分组
4. 用户修改后即时保存或显式保存

## 6.5 统一工作壳视觉要求

补充截图说明，V2 不是普通的后台表格页，而是一个“品牌区 + 工作区”的完整工作面：

- 品牌区可出现产品名、版本名、日期和设计署名
- 品牌区不放主业务操作按钮
- 导航区应与内容区形成弱分隔
- 导航选中项应具备卡片高亮
- 内容区标题层级至少包含：
  - 页面标题
  - 分组标题
  - 表单标签

这部分属于 V2 视觉基线，但不改变 `Milestone 0` 的技术优先级。

## 7. 技术架构约束

### 7.1 V2 允许新增的层

- `V2Host`：宿主挂载层，负责挂载、卸载、遮罩、关闭
- `V2App`：V2 根应用
- `stores/v2/`：只管理 V2 UI 状态，不复制业务配置
- `composables/v2/`：视图状态机、数据桥接
- `gateways/` 或同等职责层：统一读取偏好、平台配置、当前文档上下文
- `UnifiedWorkspaceShell`：快速发布与设置共用的全局壳体

### 7.1.1 历史双系统说明

当前项目存在历史分层：

- `siyuan/`：插件入口、真实思源宿主运行时
- `src/`：大量历史 SPA 页面逻辑，通过 iframe 被挂载进插件弹窗或 tab

这也是当前出现双入口、双运行时和双存储读取习惯的根本原因。

本次 V2 的总体方向必须明确为：

> 用真实 DOM 挂载逐步取代 iframe SPA，而不是继续长期维护两套前端运行形态。

### 7.1.2 构建链策略

当前构建链存在历史分工：

- V1 插件链路：以 `siyuan/` 集成为核心，保留现有 Python/esbuild 构建脚本
- 历史 SPA 链路：使用现有 `vite.config.ts`

V2 的策略应为：

- 新增独立 `vite.v2.config.ts`
- 新增独立 `dev/build` 脚本
- 直接构建 V2 的插件态入口，而不是新增 html 页面入口
- 暂时不破坏旧构建脚本
- 待 V2 稳定后，再逐步收敛和移除旧脚本

这样做的目的不是长期维持更多构建链，而是：

- 让 V2 在迁移阶段可独立验证
- 不阻断 V1 当前发布与调试
- 为后续 DOM-only 迁移提供测试通道

### 7.2 V2 禁止直接做的事

- 禁止在顶栏入口直接读 `window.localStorage`
- 禁止 V2 自己维护一套新的配置格式
- 禁止 V2 首轮重构就复制旧表单逻辑并长期分叉
- 禁止没有 Host 层就直接在入口中散写挂载逻辑
- 禁止 M0 未通过时开始写 M1 样式细节
- 禁止为 V2 新增任何 iframe 页面入口
- 禁止把旧 iframe SPA 换皮后继续称作 V2

### 7.3 配置与状态边界

#### 业务配置

业务配置继续复用现有能力：

- 平台配置
- 偏好配置
- 平台元数据
- 发布结果写回

#### UI 状态

V2 只新增界面状态：

- 当前主视图
- 当前设置子视图
- 详情面板是否展开
- 当前选中平台
- 当前发布状态

### 7.4 表单策略

首轮重构不应立即重写所有平台表单。

现状中平台设置是高度分化的：

- 单平台设置组件数量多
- 各平台存在专有字段
- 旧表单并非 schema 驱动

因此建议采用两阶段策略：

1. 第一阶段：V2 容器承接旧表单能力或桥接旧表单组件
2. 第二阶段：逐步抽象公共字段，再收敛为 schema 驱动

对于图床设置，补充截图已经给出较明确的表单布局方向：

- 页面标题在右侧内容区顶部
- 配置项按功能组分段
- 分组标题与字段标签之间保持清晰层级
- 单行字段以标签在左、控件在右的方式对齐

### 7.5 iframe 退役策略

V2 的最终目标不是“新旧 UI 永久共存”，而是：

1. 先让真实 DOM 版本承担新的主路径
2. 再逐步把旧 iframe SPA 中的高频能力迁移出来
3. 最后让 iframe 只剩历史兜底路径
4. 条件成熟后再收敛或移除

因此文档中提到的“桥接旧能力”应理解为：

- 允许复用业务逻辑
- 允许短期兼容历史能力
- 但不允许把 iframe 页面继续作为 V2 的长期承载方式

## 8. 里程碑治理规则

### 8.1 总规则

- 同一时间只允许一个活跃里程碑
- 未达到退出标准，不得开始下一个里程碑的编码
- 允许提前写设计文档，不允许提前写实现代码
- 发现当前里程碑基线错误时，优先修基线，不允许带病推进

### 8.2 验收规则

每个里程碑必须同时满足：

- 目标明确
- 允许范围明确
- 禁止范围明确
- 验收项可手工验证
- 最少有一项自动化或 smoke 验证
- 回退路径明确

### 8.3 回滚规则

当出现以下任一情况，允许并建议回滚到上一个稳定点：

- 入口行为与文档定义不一致
- V2 开关保存成功但入口不生效
- 同一配置由两套存储路径读取
- 新 UI 影响旧菜单、旧发布流程或旧设置功能
- 当前补丁已无法在一个小提交内修正

## 9. 重排后的里程碑定义

### 里程碑 0：入口与基础治理

#### 目标

建立一个真正可运行、可回退、可继续演进的 V2 入口基座。

#### 允许做的事

- 定义 `useV2UI` 开关
- 统一偏好配置读取路径
- 建立 `V2Host` 或同等职责的挂载层
- 打通顶栏主入口到 V2
- 保留旧 UI 作为回退路径
- `V2App` 可以只显示最小占位内容
- 明确 V2 主路径从第一天起采用真实 DOM 挂载
- 为 V2 新增独立构建与预览链路

#### 禁止做的事

- 直接开始写样式系统细节
- 直接开始写三栏设置页面
- 直接开始写平台卡片
- 直接开始写图床和偏好设置重构

#### 退出标准

- 开关关闭时，顶栏入口仍走旧 UI
- 开关开启时，顶栏入口进入 V2 Host，而不是旧菜单
- 旧 UI 与 V2 都读取同一份偏好配置
- 初始化失败时可自动回退旧 UI
- 至少有一份 smoke 验收清单

#### 当前状态判定

当前 `M0` 未通过。

当前问题包括：

- 开关已加，但入口行为不稳定
- 当前实现把 V2 入口接到了顶栏，但旧菜单偏好仍从另一套位置读取
- V2 挂载逻辑散落在入口文件中，未形成 Host 基座
- 文档中的入口定义和当前代码实现不一致

结论：

> 里程碑 0 必须重做或重整，未通过前禁止进入里程碑 1。

### 里程碑 1：样式隔离与基础视觉令牌

#### 目标

在 M0 通过之后，建立 V2 的样式隔离和基础 design tokens。

#### 允许做的事

- 建立 `.syp-v2` 根级命名空间
- 引入 `variables.styl`
- 引入 `base.styl`
- 对 V2 根容器、按钮、卡片、基础排版进行统一
- 为后续 `UnifiedWorkspaceShell` 预留品牌区、导航区和工作区 token

#### 禁止做的事

- 继续使用组件内部散写一套并行样式系统
- 修改旧 UI 样式
- 提前做复杂业务组件

#### 退出标准

- V2 样式通过统一入口加载
- 旧 UI 不受影响
- 样式变量可复用
- 没有同一组件既走 token 又走硬编码的双轨问题

---

### 里程碑 2：快速发布主界面框架

#### 目标

让 V2 第一屏成为真正的快速发布主界面。

#### 允许做的事

- 顶部栏
- 平台列表容器
- 空状态
- 设置入口按钮
- 读取已启用且已授权的平台名称

#### 禁止做的事

- 发布逻辑接入
- 复杂设置流程
- 详情配置表单

#### 退出标准

- 主界面能展示真实平台列表
- 点击设置按钮可以切到设置态
- 空状态和已配置态都可工作

---

### 里程碑 3：快速发布动作闭环

#### 目标

接入真实发布能力，形成 V2 的主任务闭环。

#### 允许做的事

- 平台卡片
- 发布状态
- 发布中 / 成功 / 失败反馈
- 单平台发布

#### 禁止做的事

- 全量设置重写
- 图床设置重写

#### 退出标准

- 用户可从 V2 快速发布当前文档
- 成功失败可见
- 发布失败不影响后续重试

---

### 里程碑 4：设置容器与账号列表

#### 目标

建立统一工作壳的展开态，并先打通账号列表、图床设置和平台选择，不急于全量替换旧配置表单。

#### 允许做的事

- 左侧品牌视觉区
- 左侧导航
- 账号列表
- 图床设置内容区
- 平台选择列表
- 平台配置桥接容器
- 将高频设置能力逐步从 iframe SPA 抽离到真实 DOM

#### 禁止做的事

- 一次性重写全部平台表单
- 顺带修改平台配置数据结构

#### 退出标准

- 新增账号流程可走通
- 图床设置页面可展示并承接真实配置
- 平台选择可走通
- 至少一类平台可完成桥接配置

---

### 里程碑 5：逐步替换旧设置能力

#### 目标

在前面路径稳定后，逐步替换旧设置页面中的高频能力。

#### 范围

- 偏好设置
- 图床设置
- 平台配置表单抽象

此阶段不再是“快速发布优先”的阻塞项，必须在前述里程碑稳定后再推进。

---

### 里程碑 6：收敛与稳定发布

#### 目标

完成 V2 的稳定发布策略、旧 UI 收敛策略和长期维护边界。

#### 范围

- 统计仍依赖旧 UI 的能力
- 评估哪些旧入口可以隐藏、保留或废弃
- 制定稳定发布期内的回退策略
- 形成后续废弃旧 UI 的判据
- 制定 iframe SPA 退役清单

#### 退出标准

- 至少一个稳定版本周期内无阻断性回归
- V2 主路径和高频设置路径稳定
- 旧 UI 是否继续保留有明确清单和条件
- 回退路径仍然可用

## 11. 实施依赖关系

### 11.1 硬依赖

- `M0` 不通过，`M1` 不允许开始
- `M1` 不通过，`M2` 不允许开始
- `M2` 不通过，`M3` 不允许开始
- `M3` 不通过，`M4` 不允许开始
- `M4` 不通过，`M5` 不允许开始
- `M5` 不通过，`M6` 不允许开始

### 11.2 软依赖

- 图床设置不需要阻塞快速发布闭环，但必须服从统一工作壳模型
- 平台桥接不需要在早期全量完成，但每次新增都必须保持配置兼容
- 旧表单抽象可以延后，但桥接容器必须尽早设计清楚
- 从 iframe SPA 抽离能力可以分阶段做，但不得再为 V2 增加新的 iframe 依赖

## 12. 验证与发布策略

### 12.1 验证层级

- 文档级验证：目标、边界、阶段一致
- smoke 验证：入口、开关、回退、主路径
- 页面验证：主界面态、设置展开态
- 流程验证：发布、账号新增、图床设置、偏好设置

### 12.2 发布节奏

- 先发布带回退能力的 V2
- 先开放主路径，再逐步开放设置能力
- 不以“视觉完成度”作为进入下一里程碑的理由
- 以“用户路径可用且可回退”作为核心放行条件
- 逐步让高频功能从 iframe 路径迁移到 DOM 路径

### 12.3 收敛策略

- 只有在 V2 稳定运行多个版本后，才讨论隐藏或废弃旧 UI
- 旧 UI 是否移除，必须以使用频率、回归风险和桥接完成度决定

## 14. 当前推荐动作

### 14.1 第一优先级

完成 `M0` 重整：

- 统一入口定义
- 统一偏好读取
- 抽出 V2 Host

M0 通过后再启动 `M1`：

- 完成 token 与基础基类

### 14.3 第三优先级

在 `M2-M4` 中坚持“主路径先行、设置渐进展开、桥接优先于重写”：

- 先交付快速发布主界面
- 再交付发布闭环
- 再交付设置展开态
- 最后收敛旧设置能力

### 14.4 是否允许回滚

允许。

若当前分支中的 V2 运行时代码在一个小提交内无法修通，建议回滚以下内容后重做：

- 未通过验收的顶栏挂载实现
- 未形成统一入口的 V2 运行时接线

可以保留：

- `useV2UI` 配置字段
- 文档
- OpenSpec 提案
- 截图与设计输入
